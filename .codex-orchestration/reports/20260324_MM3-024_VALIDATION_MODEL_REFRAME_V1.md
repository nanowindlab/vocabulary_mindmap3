# 20260324_MM3-024_VALIDATION_MODEL_REFRAME_V1

## Revision Meta

- 실행일시: `2026-03-24`
- 실행자: `VALIDATION_MODEL_AGENT`
- 입력 문서: `MM3-020`, `MM3-021`, `MM3-023`, `MM3-010`, `MM3-011`
- 목적: source-native 예외와 실제 결함을 구분하는 새 검증 모델 정의

## 1. 모델 목적

- 기존 검증은 “필드가 비었는가”를 먼저 보면서, source-native 부재와 실제 결함을 충분히 가르지 못했다.
- 이번 모델은 “왜 비었는가”를 먼저 분류하고, 그 다음에 “화면 계약상 문제인가”를 판정한다.
- 핵심은 결함 판정 이전에 예외 판정을 선행하는 것이다.

## 2. 예외군 정의

### 2.1 source-native 부재 예외

- 원천 데이터가 해당 값을 원래부터 갖지 않는 경우다.
- 예: 특정 항목군에서 `categories` 또는 `translations`가 source 의미상 비어 있는 상태가 자연스러운 경우.
- 이 경우는 미기입이 아니라 원천 의미의 부재로 본다.

### 2.2 source-native 대체 예외

- 같은 의미가 다른 source-native 축에 이미 존재해, 현재 필드가 비어 있어도 정보 손실이 아닌 경우다.
- 예: entry 레벨이 아니라 sense 레벨, 또는 별도 facet/sidecar에 의미가 이관된 경우.
- 이 경우는 단일 필드 기준이 아니라 대체 경로 존재 여부로 판단한다.

### 2.3 projection 비대상 예외

- 해당 필드가 그 화면의 thin projection 범위 밖에 있는 경우다.
- 예: 상세 전용 필드가 홈/결과의 얇은 렌더에서 비어 있는 것은 실패가 아니다.
- 이 경우는 화면별 계약과 분리해서 본다.

### 2.4 계산형 예외

- `sense_count`, `has_subwords`, `has_related_forms`처럼 저장형이 아니라 계산 또는 반영 결과로 해석해야 하는 항목이다.
- 저장 여부를 직접 결함으로 판정하지 않고, 계산 경로와 노출 경로를 확인한다.
- 이 항목은 빈 값 자체보다 계약된 계산 가능성의 존재가 중요하다.

## 3. 검증 대상군 정의

### 3.1 계약-핵심 대상군

- 홈/결과의 최소 식별과 탐색을 지탱하는 필드다.
- 예: `entry.id`, `entry.word`, `entry.pos`, `entry.word_grade`, `entry.categories` 요약.
- 이 대상군은 source-native 예외가 아니면 우선 검증한다.

### 3.2 의미-결속 대상군

- sense 중심 정합성을 검증하는 필드다.
- 예: `sense.id`, `definition`, `translations`, `examples`.
- 번역과 정의가 sense 단위로 결속되는지 본다.

### 3.3 화면-분리 대상군

- 홈, 결과, 상세, 표현층, 필터 바의 역할 분리가 실제로 유지되는지 검증하는 대상이다.
- thin projection과 rich detail의 경계가 섞였는지 확인한다.
- 이 대상군은 필드 존재보다 노출 위치와 깊이를 함께 본다.

### 3.4 계산-반영 대상군

- 계산형 helper가 실제 화면 또는 계약 문서와 맞게 반영되는지 검증한다.
- 저장형 테이블 존재 여부가 아니라, 계산 결과가 일관되게 나오는지를 확인한다.
- `sense_count` 계열은 여기에 속한다.

## 4. 완화 규칙

### 4.1 예외 우선 규칙

- source-native 부재가 확인되면 즉시 fail로 보내지 않는다.
- 대체 경로나 상위 층 의미가 확인되면 해당 항목은 예외 처리한다.

### 4.2 화면 한정 규칙

- 홈/결과에서 비어도 되는 필드는 상세 기준으로 fail 처리하지 않는다.
- 반대로 상세 필수 필드는 thin projection 기준으로 완화하지 않는다.

### 4.3 계산형 규칙

- 계산형 필드는 “저장 안 됨”을 결함으로 보지 않는다.
- 계산 가능한 경로가 닫혀 있으면 pass 후보, 경로가 없으면 contract risk로 분류한다.

### 4.4 잔여 결함 규칙

- 예외로 분류되지 않았고, 화면 계약상 필수이며, 대체 경로도 없을 때만 실제 결함으로 판정한다.
- 다수 누락이라도 source-native 예외가 섞여 있으면 전량 결함으로 묶지 않는다.

## 5. 실제 결함 판정 규칙

- 필드가 해당 화면의 required field인데도 source-native 예외가 아니면 결함이다.
- sense 결속이 깨져 번역이나 정의가 분리되면 결함이다.
- thin projection과 rich detail의 경계가 무너지면 결함이다.
- 계산형 필드를 저장형처럼 검사해 생긴 실패는 결함이 아니라 검증 모델 오류로 본다.

## 6. 재검증 진입 조건

- source-native 예외 목록이 먼저 정리되어야 한다.
- `categories`, `translations`, 계산형 helper가 각각 어떤 분류에 속하는지 문서상 고정되어야 한다.
- 예외를 제외한 잔여 누락이 화면별 required field 기준으로 다시 집계되어야 한다.
- 잔여 누락이 0이 아니면 재검증은 fail, 0이면 다음 단계로 진입한다.
- 동일 항목이 예외인지 결함인지 애매하면 보수적으로 예외 보류가 아니라 재분류 보류로 둔다.

## 7. 운영 판정 순서

1. 항목이 source-native 예외인지 먼저 본다.
2. 아니면 화면별 required field 대상군인지 본다.
3. 아니면 계산형 반영 대상군인지 본다.
4. 마지막에만 실제 결함 여부를 판정한다.

## 8. Reflection

- 이번 재정의의 핵심은 누락을 바로 결함으로 보지 않고, source-native 의미와 화면 계약을 분리한 점이다.
- PM redirection note처럼 `translations`와 `categories`에는 실제 결함과 source-native 부재가 섞일 수 있으므로, 단일 결론보다 분류 절차가 먼저다.
- 계산형 helper까지 저장형 결함으로 취급하면 검증이 과잉 실패로 기울기 때문에, 예외군과 대상군을 나눠 재검증 진입점을 다시 세워야 한다.
