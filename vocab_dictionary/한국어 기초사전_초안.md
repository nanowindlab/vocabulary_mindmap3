### 3-1. 이번 단계에서 확정해야 할 최소 정책
- `entry.id = target_code`를 공통 PK로 사용
- `sense.id = {entry.id}#sense-{NNN}` 형태의 생성 PK를 사용
- JSON `Sense[@val]`은 `source_ids.json`으로 별도 보존하고, API는 명시적 sense id가 없으므로 **정의문+문형+예문 시그니처 기반 매칭**을 기본 전략으로 사용
- 예문은 `texts[]` 또는 동등한 다중 발화 구조로 저장하여 대화형 예문을 보존
- 번역은 코어 DB와 분리 가능하되 `sense.id` 기준으로 역참조 가능해야 함
- 병합 기본 원칙은 **seed는 JSON, 최종 내용은 union + enrichment** 이다.
- 동일 항목이 두 소스에 모두 있을 때는 `존재 여부`가 아니라 `정보량`을 기준으로 병합한다.
  - 예: JSON에 예문이 있어도 API에 더 많은 예문이 있으면 dedupe 후 합집합 반영
  - 예: JSON에 활용형이 있어도 API에 약어형 발음이 있으면 해당 활용형 레코드에 추가
  - 예: JSON에 관련어가 있어도 API의 `link_type`/링크 정보가 더 풍부하면 함께 반영
