# 한국어 기초사전 통합 스키마 (Unified Schema v2)

---
**작성일**: 2026년 3월 23일  
**버전**: v2.3 (구현 엣지 케이스 보강)  
**분석 대상**: Open API(view) + 로컬 JSON 다운로드  
**레퍼런스 단어**: `보다` (동사/초급/61190), `하다` (동사/초급/73277), `가격` (명사/초급/62450)  
**실측 기준**: 로컬 JSON 11개 파일, 총 56,322 row / unique `target_code` 53,439 / API XML 4건 분석

---

> 현재 활성 범위는 **Phase 2-3 통합 스키마 확정**이다.  
> Phase 2-4 구축과 Phase 3 마인드맵 적용은 아직 착수 범위가 아니다.

> 이 문서의 목표는 스키마 호환이 아니다.  
> 최종 산출물은 **JSON과 Open API를 합쳤을 때 얻을 수 있는 가장 풍부한 정보량을 보존하는 집대성 자료**여야 한다.

## 3. 데이터 처리 및 병합 전략 (Phase 2 가이드)

### 3.1. 기준 PK 매핑

- 공식 API 기준 공통 ID는 `target_code`다.
  - `search` API 결과의 `target_code`
  - `view?method=target_code&q={target_code}`의 `target_code`
- 로컬 JSON의 `LexicalEntry[@val]`은 **row 고유 ID가 아니라 lexical family/group ID**로 동작한다.
- 실측 결과:
  - JSON row 56,322개
  - unique `LexicalEntry[@val]` 53,439개
  - 중복 group 1,160개 = `단어 1개 + 관용구/속담 row 여러 개`
- 따라서 구현 PK 규칙은 다음과 같다.
  - `entry.id = target_code = LexicalEntry[@val]` for grouped entry
  - 같은 `val`을 공유하는 JSON `관용구/속담` row는 독립 `entry`로 만들지 않고 `entry.subwords[]`로 접는다

### 3.2. Sense-level 매칭 규칙

> 통합 자료 구축에서 가장 중요한 난점은 **entry-level PK가 아니라 sense-level 정합성**이다.  
> `target_code`만 맞고 sense가 어긋나면 병합은 사실상 실패다.

- 통합 PK는 `senses[].id = {entry.id}#sense-{NNN}` 형태로 생성한다.
- JSON 원본의 `Sense[@val]`은 `senses[].source_ids.json`에 그대로 보존한다.
- API `sense_info`는 명시적 sense id가 없으므로 아래 순서로 매칭한다.
  - 1순위: 정규화한 `definition` 완전 일치
  - 2순위: 정규화한 `definition`의 fuzzy match 후보 중 최고점 선택
  - 3순위: `syntactic_patterns` 집합 일치
  - 4순위: 예문 시그니처(첫 문장 또는 대화 첫 발화) 일치
  - 5순위: 위 조건으로도 충돌이 없을 때만 동일 entry 내부 순서(`sense_order`) 사용
- 매칭 실패 또는 다중 후보 충돌 시 자동 병합하지 않고 `needs_review` 상태로 분리한다.

### 3.2-1. Fuzzy Matching 가드레일

- fuzzy matching은 완전 일치 실패 시에만 사용한다.
- 비교 전 정규화:
  - 연속 공백 축약
  - 문장부호/괄호류 제거
  - 조사/어미 차이로 인한 미세 흔들림을 줄이기 위한 말끝 정규화
- 허용 기준:
  - 텍스트 유사도 임계치 통과
  - 동시에 `syntactic_patterns` 또는 예문 시그니처 중 하나 이상이 보조 근거로 일치
- 금지 기준:
  - 동일 entry 내부에 유사도 상위 후보가 2개 이상 근접하면 자동 병합 금지
  - 반의/유의 관계처럼 뜻풀이가 짧고 상투적인 경우는 fuzzy 단독 승인 금지
- 구현 원칙:
  - fuzzy로 매칭된 경우 `match_method='fuzzy'` 또는 동등한 provenance를 남긴다

### 3.3. 집대성 병합 원칙

> 병합 기준은 "한쪽에 이미 값이 있으니 다른 쪽은 버린다"가 아니다.  
> 병합 기준은 **더 풍부한 구조를 흡수하고, 양쪽 정보를 dedupe 후 합집합으로 보존하는 것**이다.

- JSON은 전체 56,322 row를 빠르게 읽어 시드를 만드는 기본 소스다.
- 단, 최종 시드 엔트리 수는 `target_code` 기준 53,439개다.
- API는 빈 필드만 메우는 보조 소스가 아니라, **겹치는 필드에서도 더 풍부한 내용을 덧씌우고 확장하는 풍부화 소스**다.
- 따라서 최종 산출물은 `JSON 우선본`이 아니라 `JSON seed + API enrichment 집대성본`이어야 한다.
- 동일 의미의 필드가 양쪽에 있을 때 기본 규칙은 다음과 같다.
  - 스칼라 필드: 더 구조적이고 더 상세한 값을 채택하고, 필요하면 provenance를 남긴다.
  - 배열 필드: 정규화 후 dedupe하여 합집합으로 보존한다.
  - 보완 속성: 기존 항목을 대체하지 않고 기존 레코드에 흡수한다.
  - 충돌 필드: 자동 병합 근거가 약하면 `needs_review`로 분리한다.

### 3.4. 필드별 병합 우선순위 (v1 대비 전면 개정)

> v1의 "한국어 뜻풀이 99% 누락 → API 전수 호출 필수" 진단은 사실과 다름.  
> 다만 이것이 `JSON으로 충분`하다는 뜻은 아니다.  
> 로컬 JSON은 강한 시드이고, 최종 산출물은 **API의 풍부한 중복 정보까지 흡수한 집대성본**이어야 한다.

| 통합 대상 | 병합 기준 | 핵심 원칙 | 비고 |
|---|---|---|---|
| **한국어 뜻풀이** | JSON+API 대조 병합 | 뜻풀이가 같아도 API `reference`와 연결해 sense 단위 보강 | 둘 다 100% 보유 |
| **예문** | JSON+API 합집합 | 유형별 정규화 후 dedupe, API 추가 예문은 모두 흡수 | `문장/대화/구` 모두 대상 |
| **문형 패턴** | JSON+API 합집합 | 중복 제거 후 모두 반영 | API가 더 촘촘할 수 있음 |
| **의미/주제 범주** | JSON+API 합집합 | 같은 값은 dedupe, API 추가 범주는 흡수 | 주제 범주 확장 가능 |
| **관련어 (Sense-level)** | JSON relation + API metadata 결합 | JSON 관계어 유지, API `link_type`/링크 정보 흡수 | 관계어 자체도 합집합 |
| **관련어 (Entry-level)** | JSON related_forms + API der_info 결합 | 양쪽 항목 통합 후 링크 메타데이터 보강 | |
| **다국어 번역** | JSON 기준 유지 | 현재 JSON이 우세, 추후 API 확장 가능 시 추가 병합 | 10개 언어 일괄 내장 |
| **활용형** | JSON 활용형 + API 세부정보 결합 | API 약어형/발음/활용 상세를 기존 활용 레코드에 흡수 | 단순 대체 아님 |
| **하위 표현 (관용구/속담)** | API 우선 수용 | JSON 독립 표제어와 별도로 parent-entry 하위표현도 보존 | API 구조가 더 풍부 |
| **멀티미디어** | JSON 유지 | JSON-only 자산이므로 전량 보존 | |
| **어휘 유형 (word_type)** | API 우선 | JSON origin 유추값보다 API 직접값을 우선 채택 | API 직접 제공 |
| **원어 정보** | API 구조 + JSON 표기 결합 | `original_language.type/form` 구조를 API 기준으로 정리 | JSON은 form 보완 |

### 3.5. 풍부화 병합 규칙 예시

- `examples`
  - JSON와 API 예문을 모두 수집
  - 공백/구두점/중복 발화 정규화 후 dedupe
  - 대화 예문은 `texts[]`로 발화 경계를 유지
- `conjugations`
  - JSON의 활용형을 기본 레코드로 잡되 API의 `abbreviation_info`, 발음, 링크를 흡수
- `related_terms`
  - JSON `SenseRelation`와 API `rel_info`를 같은 대상어 기준으로 매칭
  - 매칭되면 `link_type`, `target_code`를 기존 관계어 레코드에 추가
  - 매칭되지 않으면 신규 관계어로 편입
- `subwords`
  - API `subword_info`는 parent entry 하위표현으로 보존
  - JSON에서 같은 `target_code`를 공유하는 `관용구/속담` row는 시드 단계에서 `subwords`로 접는다
  - 동일 표현이 별도 독립 표제어로도 존재하면 중복 제거하지 말고 상호 링크한다

### 3.5-1. subwords 역검색 매칭 규칙

- API `subword_info`에는 target_code가 없으므로 텍스트 기반 역검색을 수행한다.
- 단, `source_type='json_grouped'`인 subword는 이미 같은 `target_code` 그룹에서 접힌 항목이므로 역검색 대상에서 제외한다.
- 역검색 대상:
  - 코어 DB 내부 `entry.word`
  - 조건: `entry.word_unit in ('관용구', '속담')`
- 매칭 순서:
  - 1순위: 원문 문자열 exact match
  - 2순위: 공백/괄호/일부 구두점 정규화 후 normalized match
  - 3순위: 필요 시 제한적 fuzzy match
- 매칭 성공 시:
  - `subwords[].link_target_code` 기록
  - `subwords[].link_confidence`에 `exact|normalized|fuzzy` 기록
- 다중 후보 충돌 시:
  - 자동 연결하지 않고 `link_target_code=null` 유지
  - review 큐에 올린다

### 3.7. 한국어/외국어 사전 분리 구축

- `kcenter_base.seed.json`: JSON 원천 시드
- `kcenter_translations.seed.json`: JSON 번역 시드
- `kcenter_base.json`: JSON seed + API 풍부화가 병합된 최종 한국어 본체

### 3.9. 구현 수용 기준

- `target_code`와 별도로 `sense.id`가 생성되어 있어야 한다.
- 대화 예문은 발화 수가 유지되어야 하며, 문자열 1개로 합쳐 저장하지 않는다.
- JSON의 `annotation`, `syntacticAnnotation`, `Multimedia`는 손실 없이 보존되어야 한다.
- JSON에 이미 값이 있는 필드라도 API가 더 풍부하면 그 추가 정보가 실제 결과물에 반영되어야 한다.
- 예문, 문형, 관련어, 활용형은 단순 소스 우선순위가 아니라 합집합 병합이 되어야 한다.
- API 보강 결과는 `meta.source_status.api_view_fetched=true` 또는 동등한 상태값으로 식별 가능해야 한다.
- exact match 실패 시 fuzzy matching이 동작하되, 보조 근거 없는 단독 fuzzy 매칭은 허용하지 않는다.
- `subwords`는 가능하면 JSON 독립 표제어와 연결되어야 하며, 연결 신뢰도가 기록되어야 한다.
- `related_terms`, `related_forms`, `subwords`의 내부 링크는 dangling 여부가 식별 가능해야 한다.
- `target_code=0` 관계는 `is_dangling=true`이면서도 별도로 `link_status='unresolved_zero_code'`로 구분되어야 한다.
- sense-level 자동 매칭에서 충돌이 나면 강제 병합하지 않는다.
